Chapter 01

NetHack：终极 Roguelike 挑战

30年历史的经典游戏，如今成为AI研究最具挑战性的测试平台

🏰

什么是 NetHack？

1987年诞生的Roguelike地下城探险游戏。以ASCII字符渲染，每次随机生成地图，死亡即永久。目标：找到Yendor护符并返回地面。

🧩

为什么AI关注它？

数百种实体、数千条隐藏交互规则、部分可观测、长时间跨度决策。传统RL几乎无法从头学习，特别适合测试知识获取与生存决策能力。

⚡

生存压力

NetHack天然包含资源管理机制：饥饿度持续下降，不吃东西会饿死。这形成了生存压力——智能体必须在探索、战斗和资源补给之间做权衡。这正是本项目的核心场景。

🔑 与本项目的关系

NetHack的饥饿/生存机制正是我们研究的原型：智能体拥有有限的"能量"，必须决定何时花费宝贵的行动步骤去获取信息（而非直接探索或补给）。MiniHack让我们在受控环境中精确研究这一问题。

Chapter 02

MiniHack：可定制的研究沙盒

基于NLE构建的轻量级框架，用des-file快速定义自定义任务

🔧

MiniHack 是什么？

Samvelyan et al. (2021) 基于NLE构建的沙盒。用ASCII描述语言定义关卡，精确控制房间、怪物、物品布局。本项目的首选实验环境。

🎯

为什么选择它？

① 支持自定义observation_keys与action set ② 天然有资源管理/饥饿等生存语境 ③ 可快速造多难度任务 ④ NLE生态里常用PPO/RND等baseline，便于对比。

📦

观测空间

你的测试脚本已验证：glyphs(实体ID)、chars(ASCII)、blstats(27维玩家状态 — 包含饥饿度！)、message(游戏消息)。这些构成智能体的感知输入。

观测Key 形状 在本项目中的角色
glyphs_crop (9,9) 局部视野 — 感知周围实体
blstats (27,) 核心：包含HP、饥饿度等，映射为内部能量E_t
message (256,) QUERY返回的hint通过此通道注入
chars_crop (9,9) 辅助：ASCII字符版本的局部视野

💡 备选方案：AFK / Q-BabyAI

如果MiniHack工程风险过高（1周内QUERY注入不顺），可立即切换到AFK框架（Liu et al. 2022），它已实现queryable knowledge source与代码，适合快速收敛到可发表结果。

Chapter 03

强化学习基础

Agent如何通过与环境交互来学习最优行为策略

🔄 Agent-Environment 交互循环

Agent
→ aₜ →
Environment
→ oₜ₊₁, rₜ₊₁ →
Agent

每步选动作，环境返回观测+奖励。目标：最大化累积折扣奖励 Gₜ = Σ γᵏ rₜ₊ₖ₊₁

🎯

策略 π(a|s)

从状态到动作的映射。可确定性或随机。本项目使用PPO（Proximal Policy Optimization）作为固定RL主干，所有方法共享网络与超参，只改变QUERY调度策略。

💎

价值函数 V(s)

从状态s开始遵循策略π的期望累积奖励。本项目中价值函数需要同时考虑外部任务奖励和生存（能量耗尽=死亡）两个维度。

🧪

探索 vs 利用 vs 查询

传统RL只有探索和利用的权衡。本项目引入第三选择：QUERY——花费行动步骤获取外部信息。核心问题变成：何时查询最有价值？

🔑 本项目的核心创新

我们不改变RL算法本身（固定PPO），只研究信息流的结构性控制——通过内稳态门控决定何时执行QUERY动作。这让实验对照更干净：所有性能差异都归因于查询调度策略。

Chapter 04

马尔可夫决策过程 (MDP)

RL的数学基础——用五元组描述序列决策问题

M = ⟨ S, A, T, R, γ ⟩
MDP 五元组定义
符号 含义 MiniHack中的对应
S 状态空间 地图+玩家属性+物品的完整组合
A 动作空间 移动+拾取+吃+QUERY（本项目扩展）
T(s'|s,a) 转移函数 游戏物理引擎 — env.step()
R(s,a) 奖励函数 到达出口+1.0，每步-0.01，死亡-1.0
γ 折扣因子 0.99（重视长远规划）

⚠️ MDP的局限

MDP假设完全观测——但在MiniHack中玩家无法看到全部状态（隐藏属性、未探索区域）。更关键的是，本项目需要建模内部能量状态E_t——这需要扩展为POMDP。

Chapter 05

部分可观测MDP (POMDP) 与扩展

当智能体看不清世界 + 拥有内部状态——引入能量扩展POMDP

M = ⟨ S, A, T, R, Ω, O ⟩  →  扩展状态 s̃ₜ = (sₜ, Eₜ)
标准POMDP → 本项目的能量扩展POMDP
👁️

观测 ≠ 状态

智能体收到观测o（通过O(o|s)生成），同一观测可能对应不同真实状态。在MiniHack中：物品效果未知、怪物属性隐藏、视野有限。

⚡

能量扩展

本项目的关键创新：将内部能量E_t 显式嵌入状态。扩展状态 s̃=(s,E)，能量随时间衰减，吃正确食物恢复，QUERY消耗，E≤0则死亡。

🔍

信息获取价值

在POMDP中，获取信息本身有价值。本项目把这一洞察具体化：QUERY是有代价的动作，何时执行QUERY取决于内部能量状态。

Eₜ₊₁ = clip( Eₜ − c_step + g(sₜ,aₜ) − c_query · 𝟙[QUERY] )
能量动力学方程 — 终止条件：Eₜ ≤ 0 → death

🔗 POMDP → 内稳态：概念桥梁

标准POMDP关注"外部世界的不确定性"。本项目的扩展POMDP引入了"内部状态的偏离"——稳态偏差 Dₜ = max(0, E* − Eₜ)。当Dₜ大时，智能体处于"生存危机"，此时信息获取的价值更高。这正是内稳态门控的数学基础。

Chapter 06

内稳态强化学习（Homeostatic RL）

从生物学的恒温调节到AI的生存决策——内稳态如何门控信息获取

🌡️

什么是内稳态？

生物学概念：有机体维持内部环境稳定（体温、血糖）的自调节机制。Keramati & Gutkin (2014) 将其形式化：奖赏=驱动减少（drive reduction），智能体的动机来自恢复内部变量到设定点。

🎮

在RL中的应用

Homeostatic RL将内部状态（如能量、饥饿）作为奖励驱动。Yoshida et al. (2025) 强调HRRL能产生前瞻调节和风险规避——智能体不等能量耗尽才行动，而是预见性地补给。

💡

本项目的创新

既有HRRL主要把内稳态用于奖励定义。我们首次将其用于信息获取调度——用能量偏差门控QUERY动作。这填补了"生存压力下的信息价值调度"这一研究空缺。

p(QUERYₜ=1 | Eₜ) = σ( α · (E* − Eₜ − ε) )
随机门控函数 — α大→近似硬阈值，α可做消融实验

🔄 内稳态门控的信息获取循环

环境
→
观测 oₜ + 能量 Eₜ
→
门控 p(query|Eₜ)
Eₜ 低？
→ Yes →
QUERY → hint hₜ
→
π(a|o,E,m)
→

能量低时QUERY概率升高 → 获取hint帮助决策 → 做出更好行动 → 恢复能量 → 系统回到"稳态"

🧬 类比：生物体的信息获取

想象一个饥饿的动物：血糖下降（能量偏差↑）→ 触发觅食行为中的环境探查（嗅觉搜索）→ 找到食物来源 → 进食恢复血糖 → 停止搜索。本项目的门控机制正是这一生物过程的计算模型。

💬

QUERY语义（最小版本）

QUERY无参数：调用固定信息通道，返回与"当前可见/当前任务相关"的hint。避免"问什么"的语言生成，降低工程复杂度。hint作为observation额外字段注入。

📚

Hint语料库

10-50条JSON记录：
{"blue_fruit": ["蓝色果实可恢复能量"]}
含同义改写版本（用于语言shift测试）。训练用版本A，测试用版本B。

🔑 设计原则：只改变信息流，不改RL算法

所有5个baseline共享完全相同的PPO网络与超参，唯一区别是QUERY调度策略。这确保实验结果可以干净归因：性能差异 = 查询调度策略的差异。

Chapter 08

详细实验计划与安排

3个任务难度 × 5个Baseline × 消融实验 × 分布变化测试

📋 任务定义（3个难度梯度）

🟢

S1: Survival-KeyRoom（最小可跑）

小房间+门/钥匙。每步掉能量。地图有两类食物：一类补能量、一类惩罚。
目标：拿钥匙开门到达出口。
知识需求：哪种食物安全？需要QUERY获取hint。

🟡

S2: Survival-Maze（中等）

迷宫+多类可食物品(≥3种)+干扰项。出口更远。
目标：能量耗尽前到达出口。
知识需求：更多物品类型，QUERY的价值更高。

🔴

S3: Survival-Explore（可选）

程序生成多布局。训练/测试分布可分开。
目标：测试泛化能力。
知识需求：物品-效果映射可能改变。

🆚 Baseline对照（5种条件）

# 方法 QUERY策略 测试目的
1 No-QUERY 从不查询 纯RL能否解决知识匮乏问题？
2 Always-QUERY 每步都查询 无节制查询的代价有多大？
3 Periodic-QUERY 每k步查询(k∈{3,5,10}) 固定频率是否足够好？
4 Random-QUERY 随机(匹配查询次数) 门控是否优于随机？
5 Energy-Gated（本文） p(query|Eₜ)门控 能量驱动的信息获取

📊 评估指标

📈

主指标

• AUC：成功率 vs 环境步数曲线下面积
• Time-to-threshold：达到80%成功率所需步数
• 平均存活步数

🎯

信息效率

• 成功率 / QUERY次数
• 低能量命中率：P(QUERY | Eₜ<θ) vs P(QUERY | Eₜ≥θ)（验证门控是否在低能量时触发）

📐

统计检验

• 每条件 ≥10 seeds（推荐20）
• 95% CI（bootstrap over seeds）
• Mann–Whitney U 或 bootstrap 差值 CI

🔬 消融实验

消融变量 取值范围 目的
α（门控斜率） 0.5, 1, 5, 20 从软门控 → 硬阈值的效果
ε（偏移量） 0, 0.1E*, 0.3E* 门控触发点的灵敏度
c_query（查询代价） 0, 小, 中, 大 代价对查询频率的影响
B（查询预算） 0, 1, 3, 5 次/局 有限预算下的分配效率

🔄 分布变化测试（至少做2类）

🗺️

Layout Shift

训练房间参数A → 测试参数B。考察空间泛化。

🔀

Item-Effect Remap

训练时blue_fruit正确 → 测试时green_fruit正确。检验是否过拟合外观。

📝

Language Shift

同义改写hint文本。检验对提示表面形式的敏感性。
